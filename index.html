HTTP/S Протоколы

Полное исследование сетевых протоколов и моделей взаимодействия

Этот проект посвящён ключевым сетевым протоколам и моделям, которые лежат в основе современного интернета. Рассматриваются архитектурные принципы взаимодействия систем, начиная с теоретической модели OSI, переходя к практическим протоколам TCP/IP и завершая детальным разбором HTTP и его защищённой версии HTTPS.

Сетевая модель OSI

Обзор модели

Модель OSI (Open Systems Interconnection model) — базовая эталонная модель взаимодействия открытых систем, представляющая собой концептуальную модель, которая характеризует и стандартизирует коммуникационные функции телекоммуникационной или вычислительной системы независимо от их внутренней структуры и технологии.

Цель модели — обеспечение совместимости различных систем связи со стандартными протоколами. Модель состоит из семи уровней, каждый из которых использует только функции нижнего уровня и предоставляет свои функции только верхнему уровню.

Прикладной уровень (Application Layer)
Слой OSI ближайший к пользователю. Пользователи взаимодействуют непосредственно с программным приложением.
Функции включают идентификацию коммуникационных партнеров, определение доступности ресурсов и синхронизацию связи.
HTTP HTTPS SMTP DNS FTP
Уровень представления (Presentation Layer)
Преобразует протоколы и обеспечивает кодирование-декодирование данных.
Запросы приложений преобразуются в формат для передачи по сети, а полученные из сети данные преобразуются в формат приложений.
MIME XDR ASN.1 SSL TLS
Сеансовый уровень (Session Layer)
Обеспечивает поддержание сеанса связи. Устанавливает, управляет и завершает соединения между локальными или удаленными приложениями.
Обеспечивает полный дуплекс, полу-дуплекс или симплекс операции.
NetBIOS RPC SIP
Транспортный уровень (Transport Layer)
Обеспечивает надёжную передачу информации от отправителя к получателю через одну или несколько сетей, сохраняя качество сервисных функций.
Контролирует надежность линии связи с управлением потоком, сегментацией и контролем ошибок.
TCP UDP
Сетевой уровень (Network Layer)
Обеспечивает функциональные и процедурные средства передачи блока информации (датаграммы) от одного узла к другому в той же сети.
Определяет адресацию узлов и маршрутизацию пакетов по сети.
IPv4 IPv6 ICMP ARP
Канальный уровень (Data Link Layer)
Обеспечивает связь между двумя напрямую соединенными узлами (node-to-node data transfer).
Предназначен для обнаружения и исправления ошибок, которые могут возникнуть на физическом уровне.
Ethernet PPP HDLC
Физический уровень (Physical Layer)
Самый нижний уровень, определяет электрические и физические характеристики соединения данных.
Отвечает за передачу и прием неструктурированных необработанных данных на физической среде.
IEEE 802.15 RS-232 Wi-Fi Bluetooth
Модель TCP/IP
Практическая реализация сетевого взаимодействия, используемая в интернете.
Состоит из 4 уровней: Прикладной, Транспортный, Интернет (Сетевой), Сетевых интерфейсов.
HTTP/SMTP/DNS TCP/UDP IP/ICMP Ethernet/Wi-Fi

Транспортные протоколы TCP и UDP

TCP

Transmission Control Protocol — основной протокол передачи данных, обеспечивающий надежную и упорядоченную доставку данных между устройствами.

  • Работает на основе установления соединений (Three-Way Handshake)
  • Обеспечивает надежную доставку данных
  • Управляет потоком информации и подтверждает прием данных
  • Автоматическая переотправка потерянных пакетов
  • Механизмы контроля потока и перегрузки

Используется для веб-страниц, электронной почты, передачи файлов (FTP) — там, где важна надежность.

UDP

User Datagram Protocol — очень простая абстракция базовой IP-датаграммы. Является ненадежным средством передачи.

  • Ненадежная доставка без подтверждений
  • Без установления соединения
  • Пакеты могут теряться, дублироваться, менять порядок
  • Нет управления потоком данных
  • Минимальные накладные расходы (заголовок 8 байт)

Используется для DNS запросов, VoIP, онлайн-игр, потокового видео — там, где важна скорость и малая задержка.

Протокол HTTP

Основные понятия

HTTP (Hypertext Transfer Protocol) — протокол передачи гипертекстовых документов, т.е. документов, содержащих ссылки на другие документы. Этот протокол предполагает использование передачи между клиентом и сервером.

Клиент посылает запрос на сервер, последний его обрабатывает и отправляет ответ, содержащий какую-то информацию. Полученный документ может состоять из нескольких файлов.

Структура URL

http://www.asozykin.ru/courses/networks
Протокол Адрес сервера Страница

Uniform Resource Locator (URL) — единообразный определитель местонахождения ресурса. URL состоит из трех основных частей:

  • Название протокола (например, HTTP, HTTPS)
  • Адрес сервера — IP-адрес или доменное имя
  • Адрес ресурса на сервере — путь к HTML-странице, изображению, видео

Методы HTTP

Метод HTTP говорит о том, какое действие с ресурсом мы хотим совершить. Основные методы определены в RFC 9110:

Эволюция HTTP протоколов

HTTP/0.9
1991

Первая экспериментальная версия протокола, разработанная в ЦЕРН. Очень простая:

  • Только метод GET
  • Только HTML-ответы
  • Одно соединение (после передачи сразу закрывалось)
  • Около 18 рукопожатий для полного обмена
  • Нет заголовков
HTTP/1.0
1996

Первая открытая версия (RFC 1945). Существенные улучшения:

  • Добавлены методы GET, POST, HEAD
  • Появились HTTP заголовки
  • Поддержка разных типов контента (HTML, изображения, видео)
  • Одно соединение с ожиданием ответа
  • От 4 до 18 рукопожатий
HTTP/1.1
1999

Основная версия, используемая сегодня (RFC 9112):

  • Постоянные соединения для нескольких запросов
  • Все основные методы (GET, POST, PUT, DELETE и др.)
  • Обязательный заголовок Host (различает домены на одном IP)
  • Кэширование, потоковая передача, байтовые диапазоны
  • Прокси-аутентификация, улучшенное сжатие
  • От 2 до 18 рукопожатий
HTTP/2
2015

Значительное улучшение производительности:

  • Бинарный протокол вместо текстового
  • Мультиплексирование (несколько запросов в одном соединении)
  • Server Push (сервер может отправлять ресурсы до запроса)
  • Приоритизация запросов
  • Сжатие заголовков HPACK
  • От 2 до 3 рукопожатий
HTTP/3
2020

Революционное изменение транспортного уровня:

  • Использует QUIC поверх UDP вместо TCP
  • Встроенное шифрование (TLS 1.3 в протоколе)
  • 0-RTT для повторных соединений
  • Улучшенное мультиплексирование без блокировки
  • Лучшая работа при смене сети (Wi-Fi → 4G)
  • Защита от атак и потерь пакетов

Защищённый HTTPS

История и принципы

HTTPS (HyperText Transfer Protocol Secure) — защищённый протокол передачи данных, расширение протокола HTTP. Его задача — защитить информацию, передаваемую между браузером и сервером, от перехвата и модификации третьими лицами.

HTTPS был создан компанией Netscape Communications в 1994 году для браузера Netscape Navigator. В 2000 году протокол был формально определён спецификацией RFC 2818.

Сегодня HTTPS стал стандартом: Google и Mozilla продвигают его использование, браузеры предупреждают о незашифрованных соединениях, а Let's Encrypt предоставляет бесплатные сертификаты.

Принцип работы

Для перехода на HTTPS необходимо получить и подключить SSL-сертификат — специальный файл, содержащий уникальные ключи шифрования. Сертификат позволяет браузеру убедиться, что сайт подлинный.

В HTTPS используется шифрование с длиной ключа 40, 56, 128 или 256 бит. Чем больше длина ключа, тем надежнее защищены данные. Ключи являются временными и периодически меняются.

Типы SSL сертификатов

DV SSL

Domain Validation

Подтверждённый домен — самый простой тип сертификата. Верифицирует только домен.

  • Проверка: только домен
  • Протоколы: HTTPS
  • Поддомены: разрешены любые
  • Использование: блоги, личные сайты

OV SSL

Organization Validation

Подтверждённая организация — подходит для среднего бизнеса (интернет-магазины, компании).

  • Проверка: домен + организация
  • Протоколы: HTTPS + проверенный правообладатель
  • Поддомены: только включённые в сертификат
  • Использование: интернет-магазины, компании

EV SSL

Extended Validation

Расширенное подтверждение — максимальный уровень доверия (банки, платежные системы).

  • Проверка: полная юридическая проверка
  • Протоколы: HTTPS + отображение организации в браузере
  • Поддомены: только включённые в сертификат
  • Использование: банки, платежные системы

Криптографические протоколы SSL/TLS

SSL

Secure Sockets Layer — криптографический протокол, разработанный Netscape в 1995 году. Обеспечивает защищённое соединение между пользователем и веб-сервером.

Рукопожатие SSL включает три сценария: установка нового соединения, возобновление соединения и аутентификация клиента.

Сегодня прямая реализация SSL (SSL 2.0/3.0) не используется — её заменил более современный протокол TLS.

TLS

Transport Layer Security — современная версия SSL, разработанная на основе SSL 3.0.

Состоит из двух уровней: протокола Записей TLS (TLS Record Protocol) и протокола Рукопожатия (TLS Handshake Protocol).

Использует асимметричные алгоритмы для аутентификации и симметричные для сохранения конфиденциальности данных.

HTTP/3 и протокол QUIC

Протокол QUIC

QUIC (Quick UDP Internet Connections) — протокол передачи данных, разработанный Google для обеспечения быстрой, надежной и безопасной передачи данных в сети. Представляет собой эволюцию TCP.

TCP + TLS
  • Несколько RTT для установления соединения
  • Блокировка head-of-line
  • Отдельные протоколы TCP и TLS
  • Проблемы при смене сети
  • Уязвимость к атакам
QUIC
  • 0-1 RTT для установления соединения
  • Нет блокировки head-of-line
  • Встроенное шифрование TLS 1.3
  • Устойчивость к смене сети
  • Встроенная защита от атак

Ключевые особенности QUIC

Работает поверх UDP

Уменьшает задержки и ускоряет установление новых соединений. UDP не требует установления соединения как TCP и обеспечивает более гибкую передачу данных.

Встроенное шифрование

Включает шифрование на уровне протокола, что делает его более безопасным и защищенным от атак, таких как перехват и подмена данных.

Мультиплексирование

Поддерживает мультиплексирование нескольких потоков данных в одном соединении, что позволяет эффективно использовать пропускную способность сети.
